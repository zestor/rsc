# User Research Synthesis

## Thematic Analysis Process

1. **Familiarization**: Read all data before coding anything.
2. **Initial coding**: Tag each observation with descriptive codes. Be generous — easier to merge than split.
3. **Theme development**: Group related codes into candidate themes.
4. **Theme review**: Check each theme has sufficient evidence and themes are distinct.
5. **Theme refinement**: Name each theme and write a 1-2 sentence description.

For affinity mapping: one observation per note, let clusters emerge (do not pre-define categories), split clusters that grow too large, keep outliers visible.

## Triangulation

Strengthen findings by combining sources:

- **Methodological**: Same question, different methods (interviews + survey + analytics)
- **Source**: Same method, different participants or segments
- **Temporal**: Same observation at different points in time

A finding supported by multiple sources is much stronger. When sources disagree, that is signal — it may reveal different user segments or contexts.

## Interview Analysis

For each interview, extract:

- **Observations**: Behaviors (what they do) vs. attitudes (what they think/feel). Note context and flag workarounds (unmet needs in disguise).
- **Intensity signals**: Emotional language, frequency of encounter, effort spent on workarounds, consequences when things go wrong.
- **Behavior vs. stated preference gaps**: What people DO often differs from what they SAY they want. Behavioral observations are stronger evidence.

Cross-interview: track theme frequency, identify user segments with different patterns, surface contradictions (often reveal meaningful segments), find surprises that challenge assumptions.

## Survey Analysis

- Always look at distributions, not just averages (bimodal vs. normal tells different stories)
- Segment responses by user type — aggregates mask important differences
- Low response rates introduce bias; note representativeness
- Treat open-ended responses like mini interview notes: code, count themes, pull representative quotes

## Opportunity Sizing

Score each research finding on:

- **Addressable users**: How many could benefit? Use analytics, survey data, or market data.
- **Frequency**: Daily / weekly / monthly / one-time
- **Severity**: Blocker / significant friction / minor annoyance
- **Evidence strength**: Multiple sources > single source, behavioral data > stated preferences

Use ranges rather than false precision ("1,500-2,500 users monthly" not "2,137"). Show the math and assumptions transparently.

## Evidence-Based Personas

Build from behavioral patterns, not demographics or imagination:

1. Identify behavioral clusters across participants
2. Define distinguishing variables (company size, technical skill, usage frequency, primary use case)
3. For each cluster: key behaviors, goals, pain points, context, representative quotes
4. Validate by sizing each segment with quantitative data

3-5 personas is the sweet spot. A persona that does not change any product decision is not useful.

## Gotchas

- **Stated preferences are unreliable** — Users say they want feature X but never use similar features. Behavioral data always wins over self-reported preferences.
- **Small sample qualitative findings get over-generalized** — Five interviews are not statistically significant. Use qual to generate hypotheses, quant to validate at scale.
- **Personas based on demographics are useless** — "Millennial marketer" predicts nothing about product needs. Cluster by behavior, goals, and context.
- **Averages hide bimodal distributions** — A 3.5 NPS average could mean universal indifference or a polarized user base. Always check the shape.
- **Ignoring contradictions between sources** — When qual and quant disagree, it is signal. Investigate rather than choosing the more convenient source.
