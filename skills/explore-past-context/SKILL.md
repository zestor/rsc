# Explore Past Context

Retrieve and learn from past sessions and memories — the shared history between you and the user.

**Main agent**: Default to delegating exploration to a `past_context` subagent. The subagent has its own context window and returns only distilled findings — this protects the main agent's context for the actual task.

**When to read directly instead**: Only skip the subagent when you already know the exact file and the information you need — e.g., `memory_search` results already pointed to a specific memory file, or a grep hit clearly contains the answer. If you'd need to open more than one file, scan a long conversation, iterate with `memory_search`, or explore broadly, delegate.

**When to delegate**: Any time the answer isn't already visible from search results or a single known file. This includes: conversation files (even short ones can consume context), reading across multiple files, multiple `memory_search` rounds, open-ended investigations, and cross-referencing across sessions. The subagent reads, searches, and distills in its own context window, returning only the findings directly in the tool result.

### Quick sizing before deciding

- **Check turn count**: First line of `conversation.md` shows it — e.g., `# Session e29d0421 — 42 turns`.
- **Count grep hits**: If `grep -rln "keyword" past_session_contexts/` returns many files, delegate.
- **Check if memory_search is needed**: If grep finds nothing in pre-loaded data, the subagent can call `memory_search` and iterate without consuming main-agent context.

### Spawn template

```python
run_subagent(
  objective="""
  Search past sessions and memories for context about [SPECIFIC TOPIC].
  I need to know: [SPECIFIC QUESTIONS].
  """,
  task_name="explore_past_context",
  subagent_type="past_context",
  user_description="Exploring past sessions for relevant context"
)
```

Be specific in the objective: include **topic keywords** the subagent should grep for, the **timeframe** if known, and **what details** you need (code, config, decisions, errors). If you already ran `memory_search` before spawning the subagent, mention the queries you used so the subagent can skip redundant searches. If data might not be pre-loaded, tell the subagent to use `memory_search` to expand. The subagent's findings are returned directly in the `run_subagent` tool result — no need to read a separate file.

## Data Model

Past context is stored as files and directories under `past_session_contexts/`:

```
past_session_contexts/
  changelog.md                           # log of how this directory was populated, including memory_search queries
  memories/                              # distilled user facts
    {domain}/                            # grouped by topic (work/, interests/, health/, ...)
      {name}.md                          # one file per memory
  sessions/
    sessions_index.md                    # overview of all sessions
    asset_index.md                       # index of downloaded files from past sessions
    {week_dir}/
      summaries/{title}.md               # conversation summaries (last 60 days)
      {session_id}/
        conversation.md                 # full Q&A text (primary source)
        attachments/                    # user-uploaded files from this session
        ai_outputs/                     # files generated by the AI in this session
{% if has_knowledge_wiki %}  knowledge/                             # pre-distilled wiki of user context
    index.md                             # catalog of all pages
    learnings/YYYY-MM-DD.md              # dated learnings logs
    entities/                            # people, orgs, tools, services
    concepts/                            # topics, techniques, domains
    projects/                            # ongoing work, goals
{% endif %}```

Data sources: {% if has_knowledge_wiki %}**knowledge** (pre-distilled wiki synthesized across many past sessions — detailed pages on the user's work, projects, and topics; read alongside memories), {% endif %}**changelog.md** (how this directory was populated), **memories** (distilled facts), **conversation summaries** (mid-level overviews), **sessions** (full conversation history), and **assets** (files from past sessions).

### Changelog — `changelog.md`

Records how `past_session_contexts/` was populated: every preload and `memory_search` call with timestamps, result counts, and the **search queries** that were used. Useful for understanding what data is already available and which queries have already been run.

### Memory snippets — `memories/{domain}/{name}.md`

Distilled facts about the user, grouped by domain (e.g., `work/`, `interests/`, `health/`). Each file has a category header, last-updated date, and a compact summary paragraph.

```
# work.employment
Last updated: 2026-03-24

The user is an Engineering Manager at Perplexity AI.
```

These are the **densest source of user context**. One memory file can answer what would take reading 10+ session turns. Good for: preferences, background, project summaries, people, established patterns.

### Conversation summaries — `sessions/{week_dir}/summaries/{title}.md`

Mid-level overviews of past conversations, covering the last 60 days. Each file is named after the conversation title and includes a description, topics discussed, and key outcomes. Grouped by week.

These sit between memories (high-level facts) and full sessions (raw Q&A). Good for: understanding what was worked on recently, identifying which sessions to drill into, spotting recurring themes across conversations.

### Sessions — `sessions/`

`sessions_index.md` — Overview of all available sessions with timestamps and truncated Q&A previews. Each session header shows a datetime range (`YYYY-MM-DD HH:MM UTC`), and each Q&A entry is prefixed with its timestamp. Read this for orientation: which sessions exist, when they happened, and what they covered.

```
## Session `e29d0421` (2026-03-28 18:05 UTC → 2026-03-28 19:42 UTC)

- [2026-03-28 18:05 UTC] **Q:** Find daily update cron files
  **A:** Found 3 cron files related to daily updates...
- [2026-03-28 19:42 UTC] **Q:** Push the fix to the repo
  **A:** PR created: ppl-ai/asi#4012...
```

`sessions/{week_dir}/{session_id}/conversation.md` — Full Q&A for every turn. The header shows a datetime range for the session, and each turn is timestamped. The first ~15 lines are a **line-numbered, timestamped index**:

```
# Session e29d0421 (2026-03-28 18:05 UTC → 2026-03-28 19:42 UTC) — 10 turns

1. [2026-03-28 18:05 UTC] [line 16] Find daily update cron files
2. [2026-03-28 18:30 UTC] [line 50] Okay, I am assessing the quality of...
3. [2026-03-28 19:42 UTC] [line 69] Check the explore-past-context sub...
```

Each turn section also includes its timestamp:

```
## Turn 1 — 2026-03-28 18:05 UTC
### Query
Find daily update cron files

### Answer
...
```

Use the index to identify relevant turns by time and topic, then read specific line ranges. Good for: exact wording, code snippets, detailed approaches, error context, and understanding the chronological order of interactions.

### Assets — `sessions/asset_index.md`

Files from past sessions (user uploads and AI-generated outputs) are downloaded into session directories and cataloged in `asset_index.md`. Each entry shows the file type, source, caption, and exact path:

```
## my-report.pdf
- **Type:** application/pdf
- **Source:** Computer Generated
- **Caption:** Quarterly analysis report
- **Location:** sessions/2026-03-28_2026-04-03/e29d0421/ai_outputs/my-report.pdf
- **Thread:** e29d0421
```

To re-share a file from a past session: read `asset_index.md` to find the file path, then call `share_file` with that path. The files are already downloaded into the workspace — no need to copy or move them.

`knowledge/` is a pre-distilled wiki synthesized across many past sessions — entity, concept, and project pages plus dated learnings about the user. grep -rln "keyword" past_session_contexts/knowledge/ --include="*.md"

### Past briefings and articles — `knowledge_artifacts/`

Layout:

Reach for `recent.json` first; drop into per-date `feed.json` or individual `sources/*.md` only when the user asks about a specific item.

### How data gets populated

Recent sessions are pre-loaded at session start. Calling `memory_search` with targeted queries pulls in **additional** sessions and memories from the backend. If your initial search finds nothing, use `memory_search` to expand the corpus before giving up.

## Operations

Three categories of operations. These are building blocks — compose them freely in whatever order and combination gets you to the insight you need.

### Search

Find which files and lines are relevant.

**Grep across everything** (most common starting point):

```bash
grep -rn "keyword" past_session_contexts/ --include="*.md" | head -30
```

**Grep with context lines:**

```bash
grep -rn -C 3 "keyword" past_session_contexts/ --include="*.md" | head -50
```

**Find files mentioning multiple keywords:**

```bash
grep -rln "keyword1" past_session_contexts/ --include="*.md" | xargs grep -ln "keyword2"
```

**List all memory files** (useful for orientation):

```bash
find past_session_contexts/memories -name "*.md" | head -20
```

**Skim all memory headers at once:**

```bash
head -3 past_session_contexts/memories/*/*.md
```

### Read

Get the detail from files you've identified.

**Read a memory snippet** — use the Read tool:

```
Read("past_session_contexts/memories/work/projects.md")
```

**Read a conversation turn index** (first 15 lines):

```
Read("past_session_contexts/sessions/{week_dir}/{session_id}/conversation.md", limit=15)
```

**Jump to a specific turn** using line numbers from the index:

```
Read("past_session_contexts/sessions/{week_dir}/{session_id}/conversation.md", offset=50, limit=30)
```

**Read multiple memory files at once** via Bash:

```bash
cat past_session_contexts/memories/work/*.md
```

Use the **Read tool** when you need precise offset/limit control (conversation turns). Use **Bash** (cat, head) when reading multiple small files at once (memories).

### Expand

Pull in more data from the backend when pre-loaded content isn't enough.

```
memory_search(queries=["keyword or topic"])
```

This fetches additional sessions and memories into `past_session_contexts/`. After it returns, repeat your search — new files will be available. Use targeted queries for best results.

## Heuristics

{% if has_knowledge_wiki %}- **For user, work, project, and topic context, read the relevant slices of `knowledge/` and the relevant memories — not everything.** Skim `knowledge/index.md` to find which pages match, then read just those pages plus the memories that connect to the question. The two complement each other (wiki = detailed pages; memories = distilled facts) but neither is meant to be read exhaustively. Drop down to conversation summaries or sessions when the wiki + memories don't cover it, or when the user wants specific session data.
{% endif %}- **Start cheap, go deep only if needed.** Memory snippets and conversation summaries before full session reads.
- **Memory snippets often answer the question alone.** For preferences, background, or "what do you know about X" — check memories first. If they're sufficient, stop.
- **Grep is your entry point for anything keyword-based.** It searches both memories and sessions in one call.
- **Read the turn index before reading turns.** Don't read an entire `conversation.md` — scan the index, identify relevant turn line numbers, then read just those ranges.
- **Delegate to a subagent by default.** Unless the answer is already visible from search results or a single known file, spawn a `past_context` subagent. The subagent reads, searches, and distills in its own context window, returning only the findings. This keeps the main agent's context clean for the actual task.
- **Stop when you can answer the objective.** If you found the relevant context in 2 turns of one session, don't keep exploring other sessions.
- **If grep finds nothing, expand before giving up.** Call `memory_search` with relevant queries to pull in more sessions, then search again.
- **If nothing exists, say so quickly.** Don't exhaust all possible searches. If grep and one round of memory_search both return nothing, report "no relevant past context found" and stop.

## Constraints

- **NEVER glob `past_session_contexts/sessions/`.** Session IDs are opaque hashes — globbing by filename is unreliable. Search file contents instead.
